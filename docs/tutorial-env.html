Using the env

From a five-line rollout to a custom mission with red agents. Every code block on this page has been executed against this build.

1 · Hello, swarm

Environments are built from named recipes with good defaults. The simplest is "empty": an open grid, vector observations, no reward — pure physics.

import jax, kymera

env  = kymera.make("empty", grid_h=8, grid_w=8, n_agents=4)
traj = kymera.rollout(env, kymera.random_policy, 32, jax.random.PRNGKey(0))
traj["obs"].shape, traj["reward"].shape
⇒ ((33, 4, 3), (32, 4))

rollout wraps reset + step in one lax.scan. State-side entries carry a leading T+1 axis (index 0 is the pre-step state); action-side entries carry T. The whole thing is jit-pure — wrap it in jax.vmap over the key for free seed-parallelism:

keys  = jax.random.split(jax.random.PRNGKey(0), 32)
batch = jax.vmap(lambda k: kymera.rollout(env, kymera.random_policy, 32, k))(keys)
batch["reward"].shape
⇒ (32, 32, 4)   # (seeds, T, N)
Doctrine Multi-seed always goes through vmap, never a Python loop.

2 · The World

The env state is an immutable chex.dataclass pytree — it threads through jit / vmap / lax.scan unchanged. The fields are the committed contract every component (and your reward code) reads:

fieldshapemeaning
body.position(N, 2) i32agent (row, col)
body.energy(N,) f32reserved for the energy roadmap (zeros)
explored(H, W) i32per-cell visit counts (heatmaps)
seen_by(N, H, W) booleach agent's own covered cells
wall(H, W) boolobstacles
comm_graph(N, N) booledges that delivered this step
channel / missionpytreescomponent-owned state (gossip beliefs, waypoints…)
group(N,) i32team membership, assigned at reset

Two derived properties matter constantly: world.visited (explored > 0 — where anyone has stepped) and world.covered (seen_by.any(0) — the union of cover footprints). Coverage metrics read covered.

obs, state = env.reset(jax.random.PRNGKey(0))
state.body.position.shape, int(state.visited.sum()), int(state.covered.sum())
⇒ ((4, 2), 4, 4)

obs, state, reward, done, info = env.step(state, jnp.zeros((4,), jnp.int32),
                                          jax.random.PRNGKey(1))
sorted(info)
⇒ ['comm_graph', 'explored', 'metrics', 'reward_terms', 'seen_by', 'step_count']

The info keyset is fixed per env configuration, so it stacks cleanly through lax.scan.

3 · comm-coverage — the flagship recipe

Cooperative coverage with range-limited gossip: agents cover cells, share their belief maps with neighbours inside a comm radius (one-step delay, multi-hop flooding), and are rewarded for new coverage, staying connected, and not colliding. Printing the env shows the whole composition:

env = kymera.make("comm-coverage", grid=16, n_agents=4, comm_r=5, spawn_radius=2)
print(env)
⇒ GridEnv(16×16, n_agents=4, cover_r=0,
⇒   terrain=OpenTerrain(), spawn=ClusterSpawn(radius=2),
⇒   dynamics=GridDynamics(collision=NoCollision()),
⇒   channel=GossipChannel(topology=DiskTopology(radius=5, metric='chebyshev'),
⇒                         delay=1, dropout=0.0, bandwidth=None),
⇒   obs=GridObs(channels=('known', 'own_pos', 'known_walls', 'neighbors',
⇒                         'local_frontier'), sense_r=1,
⇒               central=('team_explored', 'all_pos', 'walls')),
⇒   mission=[coverage×1, connectivity×2, collision×-4])

Per-agent observations are a 5-channel grid, each channel a named entry from kymera.obs.CHANNEL_FNS:

channelwhat the agent sees
knownits post-gossip belief map (channel.shared)
own_posa one-hot at its own cell
known_wallswalls it has learned about
neighborsteammates currently inside comm range
local_frontierunknown cells within sense_r — “which way is fresh ground”

Rewards are a sum of named terms, and every term's unweighted per-agent contribution is logged each step — “which term is driving behaviour” is a curve, not an archaeology project:

traj = kymera.rollout(env, kymera.random_policy, 70, jax.random.PRNGKey(0),
                      keep="all", collect=("reward_terms",))
traj["info"]["reward_terms"].keys()
⇒ dict_keys(['collision', 'connectivity', 'coverage'])

# NOTE: on a STACKED trajectory reduce explicitly — World.covered assumes a
# single world (axis 0 is time here, not agents).
coverage = traj["world"].seen_by.any(axis=1).mean(axis=(1, 2))   # (T+1,)
f"{float(coverage[-1]):.1%}"
⇒ '24.6%'
Footgun Properties like covered / visited reduce over axis 0, which on a stacked trajectory is time. On trajectories, write the reduction explicitly, as above.
Parity That 24.6% is bit-identical to zymera v0 under the same key — the golden trajectory in tests/golden/ pins it.

4 · Composing components

Recipe kwargs cover the common knobs; for anything else, pass component objects. Both styles below were executed:

# recipe knobs: scatter spawns, hard collisions, lossy comms
env = kymera.make("comm-coverage", grid=8, n_agents=3, comm_r=3,
                  spawn_radius=None,      # None ⇒ scatter (default: clustered)
                  collision="block",      # sequential-claim: no two agents share a cell
                  dropout=0.1)            # 10% per-edge delivery failure

# full composition: an ASCII map + vector obs on a 4×8 world
from kymera.worldgen import MapFile
rooms = MapFile.from_string("""
........
...##...
...##...
........
""")
env = kymera.GridEnv(grid_h=4, grid_w=8, n_agents=2, terrain=rooms,
                     obs=kymera.obs.VectorObs())

Channel knobs deserve a note: delay sizes the gossip ring buffer (how stale relayed beliefs are), and dropout makes delivery stochastic. World.comm_graph always records what actually delivered.

Doctrine Connectivity rewards and metrics read the potential topology (who is in range) unless a term explicitly opts into delivered edges. Both are available; this keeps results comparable when you turn dropout on.

5 · Custom rewards, born in your experiment file

A reward term is a plain function (prev_world, world, action, ctx) → (N,) unweighted. Write it next to your experiment, mix it with shorthand terms, and only move it into kymera.missions_terms once it has earned its keep — the graduation rule. The simulator never blocks an idea; it just collects the winners.

from kymera import RewardTerm, metrics

def frontier_pull(prev, world, action, ctx):
    """Negative distance to the nearest uncovered cell — a gentle pull."""
    uncovered = ~world.covered & ~world.wall
    return -metrics.dist_to_frontier(world.body.position, uncovered,
                                     world.grid_h, world.grid_w)

env = kymera.make("comm-coverage", grid=10, n_agents=3, comm_r=3, spawn_radius=1,
                  terms=[("coverage", 1.0),                      # shorthand
                         ("capped_giant", 1.5, {"cap": 2}),      # parameterized
                         RewardTerm("pull", 0.05, frontier_pull)])  # yours

traj = kymera.rollout(env, kymera.random_policy, 10, jax.random.PRNGKey(0),
                      collect=("reward_terms",))
sorted(traj["info"]["reward_terms"])
⇒ ['capped_giant', 'coverage', 'pull']

Shorthand names: coverage, connectivity, collision, overlap, capped_giant, cohesion, degree, pbrs_frontier, pbrs_field, cbf_conn, cbf_coll. Terms that need shared expensive quantities declare them via requires (e.g. {"reach"}) and read them from ctx — computed once per step, and machinery you don't request never compiles.

6 · Red within blue

Adversarial agents are a mission concern, not an env fork. A GroupedMission assigns each agent a group at reset and routes each group its own mission's reward — including randomized k-of-N membership for compromise-budget threat models:

from kymera.missions import GroupedMission, Mission, RandomKofN
from kymera import missions_terms as mt

blue = Mission(terms=(RewardTerm("cov", 1.0, mt.new_coverage,
                                 mt.new_coverage.requires),))
red  = Mission(terms=(RewardTerm("disrupt", 1.0, mt.collision_count,
                                 mt.collision_count.requires),))

env = kymera.GridEnv(
    grid_h=8, grid_w=8, n_agents=4,
    channel=kymera.comms.GossipChannel(kymera.comms.DiskTopology(3)),
    mission=GroupedMission(assignment=RandomKofN(k=1),   # one hidden red, re-rolled per reset
                           missions=(blue, red)))

obs, state = env.reset(jax.random.PRNGKey(3))
state.group
⇒ Array([0, 0, 1, 0], dtype=int32)

_, _, reward, _, info = env.step(state, jnp.zeros((4,), jnp.int32),
                                 jax.random.PRNGKey(4))
sorted(info["reward_terms"])
⇒ ['g0/cov', 'g1/disrupt']

Group ids live in world.group (viz colors agents by it), and metrics/terms are namespaced per group. The action contract is unchanged — every agent still emits one movement int.

7 · Registering your own env

A recipe is a function returning a GridEnv. Registered envs get spec() / replace() round-tripping for free — which is how run provenance serializes the env:

from kymera.missions import Mission

def my_patrol(grid=12, n_agents=3):
    return kymera.GridEnv(grid_h=grid, grid_w=grid, n_agents=n_agents,
                          obs=kymera.obs.VectorObs(),
                          mission=Mission(terms=(), max_steps=50))

kymera.register_env("my-patrol", my_patrol)

env  = kymera.make("my-patrol", grid=10)
env2 = kymera.make_from(env.spec())          # {"recipe": "my-patrol", "grid": 10}
assert env.spec() == env2.spec()
env.replace(n_agents=5).n_agents
⇒ 5

Custom envs subclass nothing exotic: physics comes from the components, your mission supplies reward/done/metrics, and an optional annotations() hook returns Point / Path / Region primitives that the viz renders without knowing your mission's internals.

8 · Seeing it

from kymera import viz

env  = kymera.make("comm-coverage", grid=10, n_agents=3, comm_r=3, spawn_radius=1)
traj = kymera.rollout(env, kymera.random_policy, 25, jax.random.PRNGKey(0),
                      keep="all", collect=("reward_terms",))

viz.render_gif(traj["world"], "episode.gif", comm_radius=3)
viz.make_report(traj, "report.html", env=env, title="random policy, 10×10")

The comm overlay draws potential edges dotted and delivered edges solid — under dropout you see the difference. make_report writes a single self-contained HTML file: episode gif, coverage curve, per-term reward curves, visit heatmap, and the env composition. Agents are colored by group when groups exist.

9 · Memory profiles & the re-simulation doctrine

rollout records lean by default: the stacked trajectory drops the channel ring buffer and mission state (the first things to blow device memory at multi-seed scale). Opt back in when you need them:

callyou get
rollout(env, π, T, key) lean: positions, knowledge, walls, comm_graph — training-sized
… keep="all"full state, including gossip beliefs — viz-sized
… collect=("reward_terms",)per-step unweighted term breakdown stacked into traj["info"]
Doctrine Never archive heavy trajectories. A run is reproducible from (env spec, checkpoint, key) — re-simulate with keep="all" whenever you want to render. Delete training trajectories freely.
